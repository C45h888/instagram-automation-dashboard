## 2. What Pass 5 Discovered (and Failed to Land)

While trying to enable the IPC via `capabilities/default.json`, I ran `cargo check` and found:

**The 21 Rust commands are NOT registered as a Tauri plugin.** They are registered directly in `tauri::generate_handler![...]` in `bootstrap/runtime.rs:92-114`. This is a Phase 1-era pattern that bypasses Tauri 2.x's plugin/permission system entirely.

Tauri 2.x's permission system (the system `capabilities/default.json` queries) only knows about commands that are declared inside `tauri::plugin::Builder`. The ACL manifest at `gen/schemas/acl-manifests.json` contains only `core:app`, `core:event`, `core:menu`, etc. — Tauri's built-in plugins. There is no entry for the `automation_kernel`'s own commands because they were never wrapped as a plugin.

**Implication:** the capability identifiers like `runtime:get-state` that I tried to add in pass 5 are not valid permission identifiers. `cargo check` rejects them all with "Permission X not found, expected one of core:default, core:app:default, ..."

**The architectural fix** (NOT done in pass 5):

1. Refactor `src/ipc/commands.rs` so the commands live inside a `tauri::plugin::Builder::new("kernel")` instead of being top-level `#[tauri::command]` functions in a lib crate.
2. Add `tauri-build` plugin-generation config to `Cargo.toml` (already present per Cargo.toml:17).
3. Tauri's build script will then auto-generate `gen/permissions/kernel/autogenerated/reference.toml` files when it sees the plugin declaration.
4. Register the plugin in `bootstrap/runtime.rs` via `.plugin(tauri_plugin_kernel::init())`.
5. Add the plugin as a dependency in Cargo.toml and re-export it.
6. Update `capabilities/default.json` to reference the now-valid identifiers like `kernel:allow-runtime-get-state`.

This is a multi-file Rust refactor, not a one-line config change. Estimated scope: ~5–8 Rust files modified, plus Cargo.toml, plus capabilities, plus re-verify with `cargo check` and (ideally) a `tauri build` end-to-end test. None of this was done.

## 5. What Was NOT Done in Pass 5 (Genuinely)

- **No Rust plugin wrapper.** The `automation_kernel` crate is a lib + cdylib + staticlib (per Cargo.toml:10) but does not declare a Tauri plugin. The commands are bare top-level `#[tauri::command]` functions.
- **No `gen/permissions/kernel/` directory.** Tauri 2.x auto-generates this when it sees a plugin declaration in a Rust source file. Since no plugin is declared, this directory does not exist.
- **No `tauri build` end-to-end test.** The Rust crate compiles and `cargo check` passes, but I did not run `tauri build` to verify the bundling / capability-resolution pipeline end-to-end. This is the test that would have caught the missing plugin wrapper if I had run it earlier.
- **No capability permissions actually granted.** `capabilities/default.json` is back to `"core:default"` only — the WebView can use Tauri's built-in commands but none of the 21 kernel commands.

## 6. Why the Plan Looked Right But Wasn't

When I wrote the pass-5 spec, I extrapolated from how Tauri 1.x worked (where you could register arbitrary commands and they'd be invokable) to Tauri 2.x. I did not verify the actual ACL manifest before writing the spec, and I did not run `cargo check` until after writing the capability extension. The plan said "extend capabilities/default.json with 21 permission identifiers" — that would have worked in Tauri 1.x but is not how Tauri 2.x's plugin-based permission system works.

The TS interpretation plane (`src/lib/bridge/ipc.ts`) is the correct shape for the eventual end state. It will work without modification once the Rust plugin wrapper is added. But the Rust side needs the plugin refactor before the IPC bridge can be enabled at runtime.

## 9. Lessons Learned (For Next Session)

1. **Always run `cargo check` after modifying `capabilities/default.json`.** The npm build passes for the React/Vite side regardless of Tauri permission state — only cargo catches permission errors.
2. **Verify the ACL manifest before writing a permission extension.** `gen/schemas/acl-manifests.json` lists every valid identifier. If the command isn't there, the plugin wrapper is missing.
3. **Tauri 2.x custom commands require `#[tauri::plugin]` wrapping** to participate in the capability system. Bare `#[tauri::command]` functions registered via `tauri::generate_handler!` are Phase 1-era and don't compose with the capability system added in Phase 2.
4. **The TS interpretation plane at `src/lib/bridge/ipc.ts` is the right shape** for the eventual end state. It's preserved as-is and will become functional once the Rust plugin wrapper is added.
