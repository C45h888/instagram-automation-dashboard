Fix 1 — Stop the interval restart cascade (useAuthStore selector)
The problem at useOversightChat.ts:59 and useDMInbox.ts:133:


// CURRENT — subscribes to the entire authStore object
const { user } = useAuthStore()
When you call useAuthStore() with no selector, Zustand gives you the entire store and re-renders the component on any store change — even if user itself didn't change. The returned user object is a new reference every re-render even if user.id is identical. This matters because useDMInbox puts user?.id in a useCallback dep array:


// useDMInbox.ts:133
const fetchMessages = useCallback(async (conversationId: string) => {
  ...
}, [user?.id, businessAccountId])  // new user reference = callback recreates

// useDMInbox.ts:230
useEffect(() => {
  const intervalId = setInterval(() => fetchMessages(conversationId), 30000)
  return () => clearInterval(intervalId)  // clears old interval
}, [selectedConversationId, fetchMessages])  // fetchMessages changed = effect re-runs = interval restarts
Every time authStore ticks (token refresh, businessAccountId update, anything) → user is a new object → fetchMessages recreates → effect cleanup fires → clearInterval → new setInterval → overlapping intervals accumulate.

The fix:


// Select only the primitive value — Zustand only re-renders when user.id actually changes
const user = useAuthStore(state => state.user)
This makes user referentially stable. fetchMessages stops recreating on unrelated store changes. The useEffect only restarts when selectedConversationId genuinely changes.

Fix 2 — Eliminate duplicate post_queue query
The problem at useQueueMonitor.ts:52 and useQueueMonitor.ts:64:


// Query 1 — fires every 15s, fetches ALL post_queue rows
const statusQuery = useQuery({
  queryKey: ['queue-monitor', 'status'],
  queryFn: () => AgentService.getQueueStatus(),  // SELECT status, action_type FROM post_queue
  refetchInterval: 15_000,
})

// Query 2 — fires every 30s, fetches post_queue WHERE status='dlq'
const dlqQuery = useQuery({
  queryKey: ['queue-monitor', 'dlq'],
  queryFn: () => AgentService.getQueueDLQ(50),  // SELECT * FROM post_queue WHERE status='dlq'
  refetchInterval: 30_000,
})
The DLQ rows are already inside the statusQuery result — they're just status='dlq' rows. Two separate network requests for overlapping data.

The fix: Drop the dlqQuery entirely. Derive DLQ from statusQuery.data with a client-side filter:


// One query, one network request
const statusQuery = useQuery({
  queryKey: ['queue-monitor', 'status'],
  queryFn: () => AgentService.getQueueStatus(),
  refetchInterval: 15_000,
})

// DLQ derived client-side — zero extra requests
const dlqItems = useMemo(
  () => (statusQuery.data ?? []).filter(item => item.status === 'dlq'),
  [statusQuery.data]
)
The AgentService.getQueueStatus() query needs to select enough fields for both use cases (it currently only selects status, action_type — it would need to also select id, payload, retry_count, error etc. that DLQ currently fetches separately). The alternative is to just make getQueueStatus() select all fields once.

Fix 3 — Route agent_heartbeats through the backend
The problem at useAgentHealth.ts:53:


// CURRENT — frontend directly queries Supabase
const heartbeatsQuery = useQuery({
  queryFn: async () => {
    const result = await AgentService.getHeartbeats(5)  // SELECT * FROM agent_heartbeats
    return result.data
  },
  refetchInterval: 30_000,
})

// Then the hook manually replicates the liveness threshold:
const isAlive = heartbeat &&
  (Date.now() - new Date(heartbeat.last_beat_at).getTime()) <= LIVENESS_THRESHOLD_MS
The frontend is duplicating threshold logic that the backend already owns (oversight.js LIVENESS_THRESHOLD_MS). If you ever change the threshold, you have to update both places. Also exposes the raw agent_heartbeats table to frontend clients.

The fix: One backend endpoint that does the check and returns a clean status:


// Backend already (or should) expose:
// GET /api/instagram/agent/heartbeat-status
// Response: { status: "online" | "offline", last_seen_ms: number, agent_id: string }

const heartbeatsQuery = useQuery({
  queryFn: async () => {
    const res = await fetch(`${apiBase}/api/instagram/agent/heartbeat-status`)
    return res.json()  // { status: "online", last_seen_ms: 847000 }
  },
  refetchInterval: 30_000,
})

// No threshold math in the frontend — backend already applied the 25-min window
const isAlive = heartbeatsQuery.data?.status === 'online'
Single source of truth: the 25-min threshold lives only in oversight.js.

Fix 4 — Replace polling with Supabase Realtime
The problem: useActivityFeed, useAgentHealth (alerts), and useQueueMonitor all use refetchInterval — they pull on a timer whether or not anything changed. Supabase Realtime pushes only when a row is actually inserted/updated.

The pattern:


// CURRENT — blind pull every 30s
const { data } = useQuery({
  queryFn: () => AgentService.getAuditLog(50),
  refetchInterval: 30_000,
})

// AFTER — initial fetch + push updates only
const [entries, setEntries] = useState([])

useEffect(() => {
  // Fetch initial data once
  AgentService.getAuditLog(50).then(r => setEntries(r.data ?? []))

  // Subscribe to new inserts — only fires when agent writes to audit_log
  const channel = supabase
    .channel('audit-log-live')
    .on(
      'postgres_changes',
      { event: 'INSERT', schema: 'public', table: 'audit_log' },
      (payload) => setEntries(prev => [payload.new, ...prev].slice(0, 50))
    )
    .subscribe()

  return () => supabase.removeChannel(channel)
}, [businessAccountId])
Same pattern applies to system_alerts (INSERT/UPDATE) and post_queue (UPDATE on status changes). The audit_log table is the best candidate — it's append-only (INSERT only), which is the cleanest Realtime use case.

Fix 5 — Remove the dead realtimeService 3s poller
The problem at realtimeService.ts:85:


// Fires every 3 seconds, hits backend /webhook/realtime-updates
this.pollingInterval = setInterval(async () => {
  const response = await axios.get(`${API_BASE_URL}/webhook/realtime-updates`)
  // ... data returned here but nothing in the dashboard consumes it
}, 3_000)
3 seconds is the most aggressive interval in the whole frontend. The data it fetches isn't wired to any component state in the Agent Terminal Dashboard. It's pure network noise — 20 requests/minute for nothing.

The fix: Either wire it to a real consumer or delete realtimeService.ts entirely and replace with Fix 4 (Supabase Realtime subscriptions). There's no middle ground — a 3s poller that nobody reads is just overhead.
